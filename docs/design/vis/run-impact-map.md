# Run Impact Map

用户看完一次 agent run，最想知道的经常不是“所有事件”，而是：

> 这次 run 的影响范围有多大？

Run Impact Map 不是安全告警页，也不是文件系统图。它把一次 run 的影响聚成几个区域：workspace、process、network、resource、tests、generated output、unattributed activity。

每个区域都必须能连回语义来源。

## 适合什么场景

- run 结束后的 summary
- PR review 的第一屏
- 判断 agent 是否只围绕用户目标行动
- recovery 前快速知道需要检查哪些地方
- 比较两个 run 的行为范围

## 视觉结构

```text
Goal: "fix failing API test"

Impact Radius
├─ Workspace
│  ├─ read: tests/api.test.ts     <- inspect failure
│  ├─ read: src/api/handler.ts    <- patch handler
│  └─ write: src/api/handler.ts   <- patch handler
├─ Processes
│  ├─ npm -> node -> jest         <- reproduce failure
│  └─ npm -> node -> jest         <- verify fix
├─ Network
│  └─ registry.npmjs.org          <- npm test dependency lookup
├─ Resources
│  ├─ 13s wall time
│  └─ 1380 tokens
├─ Results
│  ├─ first test: exit 1
│  └─ second test: exit 0
└─ Unattributed
   └─ git status x6
```

## 视觉编码

- 中心是 user goal 或 run label。
- 周围是 impact zones。
- 每个 impact item 旁边显示 semantic source。
- item 大小表示 effect weight，例如 writes、network、duration、cost、retry count。
- 没有 semantic source 的 item 进入 `Unattributed` 区域。

## 和 Workspace Map 的区别

Workspace Map 只关心 repo/workspace 内部行为。

Run Impact Map 关心完整影响范围：

- workspace 改动
- process tree
- network destination
- token/cost/resource
- tests and commands
- generated output
- background activity

它适合作为“这次 run 碰了哪些东西”的第一张 summary 图。

## 不要做成纯计数面板

下面这种 summary 不够：

```text
17 files touched
4 processes
2 network calls
1380 tokens
```

它缺少因果关系。

更好的写法是：

```text
17 files touched
  12 read while locating parser fallback
  3 written during patch phase
  2 generated by test runner

4 processes
  npm/node used to reproduce failure and verify fix

2 network calls
  registry lookup triggered by npm test
```

Run Impact Map 的重点不是数量，而是“影响范围是否和语义目标相称”。
